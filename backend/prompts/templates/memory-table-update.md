# 规则
1. 只输出一个 JSON 数组，每个元素是一个操作。除 JSON 外不输出任何字符。
2. 操作只有四种：
   - `{"table":"<表key>","op":"add","row":{列:值,...}}` 新建行（不要写 id，系统自动分配）
   - `{"table":"<表key>","op":"update","id":<行id>,"fields":{列:值,...}}` 改已有行（id 必须照抄上方表中真实存在的 id）
   - `{"table":"<表key>","op":"close","id":<行id>,"reason":"..."}` 归档退场的行（NPC 死亡、物品消耗、或定局表里某事实被 retcon 撤销），reason 会被记进归档
3. 只用上方「表结构」列出的 key 和列名，写错的列会被丢弃。
4. **新建前必须先查重**，避免重复行：
   - relations 这种没有单一主名的表：按全部关键列判重（用「主体A+主体B+关系类型」，且 A、B 按字典序排列固定方向）。
   - 其余表（含势力表，按「势力」名）：按首列实体名和「别名」列判重。
   - 若实体已存在（哪怕换了称呼），用 update 改它并把新称呼追加进「别名」字段，不要新建重复行。
5. **写对表、对准主语**：每张表的「用途」说明了它的记录视角。地点表只记跟这个地方绑定的客观事实（归属、物理状态、危险/资源），主语永远是地点；角色之间的行动、阴谋进展、谁对谁做了什么属于关系表（关系变化）或历史召回（事件叙事），不要把剧情流水写进地点表的「当前状态」「历史标记」。势力表每行钉在一个「有组织的群体」上（不限题材：门派/家族/朝廷/商会，也含公司/机构/团伙/线上群组等，凡「一群人的集体」皆算），记它当前的范围、实力、立场等静态快照；势力之间的立场写在势力表「立场」列，不重复进关系表。**正文里只要出现这样一个群体就为它在势力表立行，即使它的成员已经在关系表里也要立——组织和成员关系并存，不算重复。**不要把事件经过写进势力表。定局表门槛极严、宁缺勿滥（只留极少数行）：**只收五类不可逆硬事实——① 死亡/永久退场 ② 身份或真相被揭穿 ③ 永久性损毁/封印/摧毁 ④ 立下的誓约/契约/不可反悔的承诺 ⑤ 重大且不可逆的获得或失去；不在这五类里的（喝水、开门、对话、赶路、心情、临时移动、未达成的目标、悬念）一律不记。**且必须能在「红线」列写出"这条事实禁止后续怎么写"，**写不出具体被禁的写法就不准 add**。「既定事实」一句话名词化、「红线」写它禁止了什么。**定局表的行只在剧情回滚或改设定（retcon）让某事实不再成立时才 close（reason「设定撤销」），正常推进永不 close。**定局表只放上述五类红线，事件细节归历史召回；同一件事两边都有不算重复。
6. 字段值精炼：只写结果不写过程，**每个字段 ≤60 字**（超出会被截断），「历史标记」「产出/收益」等控制在一句话内、名词化标记，不要叙事。
7. 只有发生**实质变化**才动表；闲聊、无信息推进时不输出任何操作（空数组 `[]` 即可，不需要为没动的表写 noop）。
8. **行数上限（见上方「各表行数上限」）**：某张表标「已满」或新增后会超过上限时，每要 `add` N 行，必须先对这张表 `close` 掉 N 行**最不重要**的旧行（最久没被提及、已无剧情价值、过期或已被取代的），再 add。务必腾够空间，让该表新增后不超过上限。不限制的表无需归档。归档优先级由你判断，不要机械地删最早那行。

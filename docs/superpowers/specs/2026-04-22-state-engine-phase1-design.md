> 日期：2026-04-22  
> 范围：触发器系统（数据层 + 执行时机 + 动作副作用 + 前端 UI）  
> 背景：v2 整体分四阶段（State → 时间线 → Forge → Story 双预设），本文件覆盖 Phase 1。

## 一、整体分阶段策略

| 阶段 | 内容 | 说明 |
|---|---|---|
| **Phase 1（本文件）** | 触发器系统 | 核心锚点；角色/世界状态字段已有，无外部依赖 |

## 二、数据层

### 新增三张表

**`triggers`**

| 字段 | 类型 | 说明 |
|---|---|---|
| id | TEXT PK | `crypto.randomUUID()` |
| world_id | TEXT FK | 所属世界 |
| name | TEXT | 触发器名称 |
| enabled | INTEGER | 0/1 |
| one_shot | INTEGER | 0/1；触发后自动设 enabled=0 |
| last_triggered_round | INTEGER | 上次触发轮次，NULL 表示从未触发 |
| created_at | INTEGER | `Date.now()` |

**`trigger_conditions`**

| 字段 | 类型 | 说明 |
|---|---|---|
| id | TEXT PK | |
| trigger_id | TEXT FK | 所属触发器 |
| target_field | TEXT | 格式 `实体名.字段名`，如 `凛.好感度`、`世界.戒严等级` |
| operator | TEXT | 数值：`>` `<` `=` `>=` `<=` `!=`；文本：`包含` `等于` `不包含` |
| value | TEXT | 条件值（按字段类型解释） |

同一触发器的多个条件为 **AND 关系**，全部满足才触发。OR 场景用两个独立触发器实现。

**`trigger_actions`**

| 字段 | 类型 | 说明 |
|---|---|---|
| id | TEXT PK | |
| trigger_id | TEXT FK | 一个触发器对应一个动作 |
| action_type | TEXT | `activate_entry` / `inject_prompt` / `notify` |
| params | TEXT | JSON，格式见下方 |

**params 格式按 action_type：**

```json
// activate_entry
{ "entry_id": "xxx" }

// inject_prompt
{
  "text": "城市进入高度戒备，NPC 行为应更谨慎",
  "mode": "consumed",       // "consumed" | "persistent"
  "inject_rounds": 3,       // mode=consumed 时有效
  "rounds_remaining": 3     // 运行时倒计时，归零后停止注入
}

// notify
{ "text": "表白剧情已解锁" }
```

### `prompt_entries` 表新增字段

| 字段 | 类型 | 说明 |
|---|---|---|
| position | TEXT | `system`（注入系统提示词区）/ `post`（注入后置提示词区）；默认 `post` |

### `one_shot` 与 `consumed` 的关系

两者是独立维度：
- `one_shot` 控制触发器本身是否只执行一次（执行后 enabled=0）
- `consumed` 控制 inject_prompt 动作的注入轮数

组合示例：触发器 one_shot=1 + inject_prompt consumed 3轮 = 触发器只执行一次，但注入内容持续 3 轮。

## 三、执行时机与异步队列集成

**触发时机**：Priority 2 状态更新落库后、Priority 3 turn record 入队前，同步执行触发器检查。

**执行流程：**

```
AI 回复结束
  → Priority 2：角色 / 玩家 / 世界状态更新（已有）
  → [新增] 触发器检查（同步）
      读取当前最新状态
      逐一评估所有 enabled triggers（同一 world_id）
        所有 conditions 满足 →
          执行 action（见下节）
          更新 trigger.last_triggered_round = 当前轮次
          one_shot=1 时设 trigger.enabled = 0
  → Priority 3：createTurnRecord（已有）
```

**触发器检查为同步操作**，不入队，避免与 Priority 3 产生竞争条件。

**inject_prompt 的 rounds_remaining 递减时机**：在 assembler.js 组装时检查，每次成功注入后 `rounds_remaining - 1`，归零后不再注入。

## 四、动作副作用

### `activate_entry`

将 `prompt_entries` 表中目标条目的 `enabled` 设为 1。持续生效直到用户手动关闭。复用现有 Prompt 条目机制，条目在 State 页四分区中可见并显示为已激活状态。

### `inject_prompt`

- **consumed 模式**：触发后在 `trigger_actions.params` 中记录 `rounds_remaining = inject_rounds`。assembler.js 按 position 字段决定注入位置（system / post）；每次注入后 `rounds_remaining - 1`，归零停止；若触发器再次满足条件可重新触发（`rounds_remaining` 重置）。
- **persistent 模式**：触发后持续注入，直到用户手动禁用触发器。

### `notify`

触发器检查完成后，通过现有 SSE 管道发送 `trigger_fired` 事件，payload 包含触发器名称和通知文本。前端监听后在对应轮次消息底部显示 `◆ 触发：XXX` 徽章，点击可跳转 State 页触发器详情。

## 五、前端 UI

### 世界详情页：三标签导航

顶部新增三个标签页：**构建 · 故事 · 状态**。

- **故事**：对应现有角色列表 + 会话入口（现有功能，路由不变）
- **状态**：新增（Phase 1 实现）
- **构建**：占位（Phase 3 实现）

导航结构一次到位，避免后续再改页面骨架。

### State 页：四分区布局

```
┌─────────────────────────────────────────────────────────┐

│  │ ● 世界背景概述  [系统提示词]  [编辑] [删除]       │   │

├─────────────────────────────────────────────────────────┤
│  🔑 关键词触发条目                    [+ 新建]          │

│  ┌──────────────────────────────────────────────────┐   │
│  │ ● 暗影帮内情  触发词：暗影帮/影堂  [编辑] [删除] │   │
```

**条目注入位置**：每个条目（常驻 / 关键词 / AI召回）编辑时可选择注入位置：
- `系统提示词`：注入 assembler [1-4] 系统区
- `后置提示词`：注入 assembler [15] 末尾

### 触发器编辑弹窗

三个区域：
1. **基本信息**：名称、启用 toggle、仅触发一次 toggle
2. **条件列表**：每行 `[目标字段▼] [操作符▼] [值]`；目标字段从当前世界所有状态字段下拉选；可添加多行（AND），可删除行
3. **动作**：单选动作类型，对应表单：
   - `activate_entry`：从当前世界条目列表下拉选
   - `inject_prompt`：填文本 + 选模式（consumed/persistent）+ consumed 时填注入轮数
   - `notify`：填通知文本

### 世界设置页简化

条目管理全部迁移到 State 页后，世界设置页只保留：
- 世界名称、世界简介（纯展示，不注入 prompt）
- LLM 参数配置（model、temperature 等）

## 六、未解决的问题（实施前需确认）

1. **assembler.js 段位**：inject_prompt 的 system 位置具体插入哪个段号（[1]-[4] 中），post 位置是否复用 [15] 还是新增段位，需要阅读当前 assembler.js 后决定
2. **条目 position 字段迁移**：现有 prompt_entries 数据需要设置默认 position 值（`post`），避免迁移后行为变化
3. **触发器检查的 session 上下文**：触发器检查时需要知道"当前轮次"，需要确认 session_id 和 round_index 如何传入检查函数
